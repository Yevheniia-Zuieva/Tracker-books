# TRACKER BOOKS

**Tracker Books** – це сучасна вебплатформа для менеджменту особистої бібліотеки. Додаток допомагає організувати процес читання та аналізувати власну активність.

### Основні можливості
* Відстежувати прогрес читання.
* Додавати нові книги до списків.
* Керувати статусами прочитаного.
* Аналізувати читацьку активність.

---

### Попередні вимоги
Перед початком переконайтеся, що у вашій системі встановлено:
- **Python 3.12+**
- **Node.js 20+** та пакетний менеджер **npm**
- **Git** (для клонування репозиторію)

## Стек технологій

### Backend
* **Python 3.12+** & **Django 6.0.2**
* **Django REST Framework 3.16.1** – для побудови API.
* **Djoser** & **SimpleJWT 5.5.1** – автентифікація користувачів.
* **Pillow 12.1.0** – обробка зображень.
* **Django CORS Headers 4.9.0** – налаштування доступу для фронтенду.

### Frontend & UI
* **React 19.x** (з використанням **Vite 7.x** для швидкої збірки).
* **Tailwind CSS 4.x** – сучасна стилізація компонентів.

### Інше
* **База даних:** SQLite (легка та швидка для розробки).
* **Тестування:** Django TestCase (Unit та API-тести).

## Архітектура системи
Проєкт реалізовано за архітектурою розділеного клієнта та сервера (Decoupled Architecture):

1. **Web Server & Frontend:** React (Vite) – забезпечує інтерфейс користувача.
2. **Application Server:** Django REST Framework – обробляє бізнес-логіку та запити до БД.
3. **СУБД:** SQLite (Development) – локальне сховище даних.
4. **External API:** Google Books API – інтеграція для пошуку метаданих книг.

## Встановлення та запуск

### 1. Налаштування Backend
```bash
cd backend

# 1. Створення та активація віртуального середовища
python -m venv venv
source venv/bin/activate  # для macOS/Linux
venv\Scripts\activate     # для Windows

# 2. Встановлення залежностей
pip install -r requirements.txt
```
Налаштування оточення. 
Створіть файл .env у папці backend/ та додайте ваші ключі::

```bash
SECRET_KEY=your_secret_key
DEBUG=True
GOOGLE_BOOKS_API_KEY=your_api_key
EMAIL_USER=your_email
EMAIL_PASS=your_password
```
Запуск бази даних та сервера:
```bash
python manage.py migrate
python manage.py runserver
```
### 2. Налаштування Frontend
```bash
cd frontend
npm install
npm run dev
```

### База даних
Проєкт використовує SQLite, що не потребує встановлення окремого сервера БД. 
Для ініціалізації файлу бази даних (`db.sqlite3`) та створення структури таблиць виконайте:
```bash
python manage.py migrate
```

Для запуску тестів бекенду:

### Корисні команди для розробника
- **Створення міграцій** (після зміни моделей): `python manage.py makemigrations`
- **Створення суперкористувача** (для доступу в /admin): `python manage.py createsuperuser`
- **Перевірка стилю коду** (Linter): `ruff check .` (backend) або `npm run lint` (frontend)
- **Збірка фронтенду для продакшену**: `npm run build`
- **Резервне копіювання бази даних:** оскільки використовується SQLite, достатньо створити копію файлу `db.sqlite3`.
- **Експорт даних у JSON:** `python manage.py dumpdata > backup.json`

## Правила документування (Contributing Guidelines)

Документування коду є невіддільною частиноюпроцесу розробки. Щоб підтримувати проєкт у чистоті та зробити його зрозумілим для всіх учасників команди, будь-які нові функції, моделі чи компоненти повинні бути задокументовані до того, як код потрапить до головної гілки.

### Backend (Python / Django)
Офіційним стандартом для написання коментарів у бекенд-частині проєкту є **Google Style Python Docstrings**. 
* **Моделі даних:** обов'язково описуйте загальне призначення кожної нової моделі та її ключові атрибути.
* **Методи та API Views:** усі нестандартні методи та View-класи повинні мати опис. Обов'язково зазначайте інформацію про вхідні аргументи (`Args`) та структуру даних, що повертається (`Returns`).
* **Генерація OpenAPI:** зверніть увагу, що якісні коментарі у `ViewSets` є критично важливими, оскільки на їхній основі автоматично генеруватиметься OpenAPI (Swagger) документація.

### Frontend (React / JavaScript)
Для опису клієнтського коду ми використовуємо стандарт **JSDoc**.
* **React-компоненти:** кожен новий компонент повинен супроводжуватися коментарем із описом його призначення та переліком усіх вхідних пропсів (`@param`).
* **Бізнес-логіка:** вимагається додавати пояснення до складної бізнес-логіки, кастомних хуків та загальних утилітарних функцій, щоб спростити їхнє подальше використання іншими розробниками.
