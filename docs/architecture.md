# Технічна архітектура та бізнес-логіка

## 1. Архітектурні рішення
Проєкт базується на розділенні відповідальності (Separation of Concerns):
- **Django REST Framework (Backend):** керує збереженням даних, автентифікацією (JWT) та валідацією.
- **React (Frontend):** керує станом інтерфейсу та відображенням даних.

## 2. Взаємодія компонентів
Фронтенд взаємодіє з бекендом через REST API. 
**Важливо:** для полів моделей використовується `camelCase` (наприклад, `totalPages`), що дозволяє уникнути трансформації ключів при передачі JSON між React та Django.

## 3. Складні алгоритми та Бізнес-логіка
### Алгоритм розрахунку прогресу читання
Прогрес обчислюється автоматично при кожному оновленні поточної сторінки книги:
- **Вхідні дані:** `currentPage`, `totalPages`.
- **Логіка:** Якщо `currentPage` досягає `totalPages`, статус книги автоматично змінюється на `read`.
- **Формула:** $\text{progress} = \frac{\text{currentPage}}{\text{totalPages}} \times 100$

## 4. "Жива" документація та сценарії тестування
Проєкт використовує підхід **Test-Driven Documentation**. Тести є не лише засобом контролю якості, а й актуальною документацією з прикладами використання.

### Модульні тести (Unit Tests)
Знаходяться у `tracker.tests.test_unit`. Вони документують внутрішню логіку:
- **Валідація сторінок:** перевірка того, що кількість прочитаних сторінок не може перевищувати загальну кількість сторінок у книзі.
- **Розрахунок мети:** приклади того, як система обчислює прогрес виконання річної мети читання.

### Інтеграційні тести API
Знаходяться у `tracker.tests.test_api`. Вони служать прикладами для фронтенд-розробника:
- **Авторизація:** тест `test_06_login_success` показує формат отримання JWT-токенів.
- **Взаємодія компонентів:** тест `test_10_add_session_updates_book_status` демонструє, як створення сесії через API впливає на статус книги.
- **Зовнішні сервіси:** тест `test_08_external_search_integration` описує протокол обміну даними з Google Books API.
