# Звіт про аналіз продуктивності системи (Performance Audit)

## 1. Методологія тестування
Для об'єктивної оцінки швидкодії системи було проведено стрес-тестування на великому масиві даних та профілювання ключових сценаріїв взаємодії.

**Інструментарій:**
* Backend Profiling: Django Debug Toolbar 6.0.2 (аналіз SQL-запитів та сигналів).
* Frontend Profiling: Chrome DevTools (вкладка Network) для вимірювання Time to First Byte (TTFB).
* Тестове середовище: локальний сервер розробки, БД SQLite/PostgreSQL.
* Обсяг даних: 50 користувачів, 1000 книг, 5000 пов'язаних об'єктів (сесії читання, нотатки).

## 2. Результати вимірювань (Baseline Metrics)
Дані отримані до впровадження оптимізацій при завантаженні повної бібліотеки користувача (1000 записів).

### Сценарій А: Автентифікація та профілювання
* JWT Token Creation (/create/): 1190 ms
* Profile Retrieval (/profile/): 52 ms
* SQL Queries: 1 запит (норма).

### Сценарій Б: Завантаження списку книг (The Library View)
* Загальний час відповіді (Total Time): 1444.28 ms (1.44 сек).
* Час виконання SQL (SQL Time): 123.67 ms.
* Затримка серіалізації (Serialization Overhead): ~1320 ms.
* Кількість SQL запитів: 1001 запит (критична аномалія).

## 3. Виявлені проблеми з продуктивністю (Hotspots)
На основі отриманих метрик ідентифіковано три ключові "гарячі точки":

### 3.1. Критична проблема N+1 Select
Найбільш вагома точка деградації продуктивності.

* **Суть:** при отриманні списку з 1000 книг, ORM виконує 1 основний запит і 1000 додаткових запитів для отримання вкладених сесій читання (reading_sessions).
* **Доказ:** панель SQL зафіксувала 1000 ідентичних (similar) запитів.
* **Наслідки:** експоненціальне зростання часу відповіді при збільшенні кількості книг.

### 3.2. Навантаження на CPU під час серіалізації
* **Опис:** понад 90% часу запиту витрачається на рівні Python-коду (Django Rest Framework).
* **Причина:** обробка 1000 вкладених об'єктів у циклі без використання методів "жадібного завантаження".
* **Наслідки:** блокування потоку виконання сервера, низька пропускна здатність системи.

### 3.3. Ефективність пошуку (Full Table Scan)
* **Опис:** пошук за параметром ?search=Книга ініціює повне сканування таблиці.
* **Доказ:** аналіз плану виконання (Explain Plan) показав тип доступу Seq Scan (Sequential Scan).
* **Наслідки:** відсутність індексів на текстових полях призведе до критичного сповільнення пошуку при накопиченні великої кількості даних.

### 3.4. Оверхед синхронних сигналів
* **Опис:** виявлено 40 активних отримувачів сигналів (Signal Receivers) на кожну дію з моделлю.
* **Наслідки:** прихована затримка (~20ms) на кожну операцію запису, що виконується синхронно.
